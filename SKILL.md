# Project Wiki Bootstrap

## 目标

在最短时间内建立一份可复用的项目 Wiki，帮助自己和团队快速回答以下问题：

- 这个项目做什么
- 怎么跑起来
- 核心代码在哪
- 关键依赖与边界是什么
- 常见风险和排查入口是什么

## 使用模式

先判断用户需要哪种深度：

- `15 分钟速览`：只做静态扫描，输出项目地图、启动路径、风险和待确认。
- `1 小时深挖`：补充关键链路、配置流向、测试/构建验证。
- `团队 Wiki`：更重视可读性、术语解释、交接路径和长期维护。
- `开发前摸底`：更重视模块边界、变更落点、验证命令和风险。

如果用户没有指定，默认使用 `15 分钟速览 + 可继续深挖`。

## 执行原则

- 先“可运行”，再“可解释”，最后“可演进”
- 结论必须可追溯到仓库证据（文件、脚本、配置）
- 优先输出高价值信息，避免堆砌低价值细节
- 不猜测未知内容，未知项明确标记为 `TODO/待确认`
- 对每条关键结论给出“证据来源 + 置信度（高/中/低）”
- 不暴露密钥、私有配置值、客户数据或业务敏感样例；只描述变量名、文件路径和用途

## 首次输入（建议）

调用本技能时，优先补充以下信息以提升准确度：

- 目标受众（自己上手 / 团队共享 / 新人培训）
- 目标深度（15 分钟速览 / 1 小时深挖）
- 目标输出路径（如 `docs/PROJECT_WIKI.md`）
- 是否要求“必须可运行验证”或“仅静态梳理”

## 证据格式（强制）

关键结论统一使用以下格式：

- 结论：
- 证据：`path/to/file`（必要时补充符号名）
- 置信度：高/中/低

## 执行步骤

1. 建立项目快照
   - 识别技术栈、包管理器、构建系统、主语言
   - 定位入口文件、主配置、环境变量样例、启动脚本
   - 扫描目录，划分核心模块与非核心模块

2. 打通最小运行路径
   - 记录安装依赖、启动命令、测试命令、构建命令
   - 明确运行前置条件（数据库、缓存、外部服务、密钥）
   - 形成“从 clone 到运行”的最短步骤
   - 若工具链缺失（如无 cmake/msbuild），明确记录“未验证原因 + 替代路径”

3. 梳理架构与数据流
   - 标出请求入口、核心业务层、数据访问层、外部依赖
   - 说明主要模块职责和边界
   - 给出 1-2 条关键业务链路（输入 -> 处理 -> 输出）

4. 提炼开发者高频问题
   - 新功能加在哪里
   - 新接口如何接入
   - 配置改动会影响哪些模块
   - 常见报错如何定位（日志、断点、关键命令）

5. 形成可维护 Wiki
   - 输出标准结构文档
   - 对不确定信息打标，列待确认清单
   - 给出下一轮补全文档建议

6. 执行后复盘（必须）
   - 给出“本次 Wiki 产出中的信息盲区”
   - 给出“skill 自身改进建议（3 条内）”
   - 若已修改技能模板，说明变更点

## 项目类型加速清单

根据项目形态追加最小检查项：

- Web/API 项目：启动端口、健康检查接口、鉴权方式、配置注入点
- SDK/库项目：导出符号、ABI/架构兼容、示例调用路径
- 桌面/工具项目：入口参数、日志目录、依赖运行时
- 嵌入式/总线项目：工具链版本、示例配置、脚本触发路径、位数匹配

## 质量门槛（完成定义）

完成时应满足：

- 新成员可在 10-20 分钟内跑起项目
- 能定位至少 3 个核心模块及其职责
- 至少包含 1 条端到端关键链路说明
- 列出明确的风险与待确认项，不留“隐性未知”
- 至少给出 5 条“结论-证据-置信度”三元组
- 若无法运行验证，必须说明阻塞点与替代验证结果

## 输出风格

- 使用短句和列表，减少长段落
- 术语统一，首次出现给出解释
- 命令可直接复制执行
- 避免贴大段源码，优先链接与总结
